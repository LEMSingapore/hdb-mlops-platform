# ADR 0007: CI workflow and how CI gets a champion to test against

**Status:** Accepted
**Date:** 2026-06-16

## Context

Sessions A and B containerised the platform: a multi-stage FastAPI image ([ADR 0005](0005-multi-stage-dockerfile-with-uv.md)) and a docker-compose stack with MLflow running as its own service ([ADR 0006](0006-mlflow-tracking-server-as-compose-service.md)). The stack came up clean for a manual smoke test and returned a Tampines prediction. Session C makes that proof automatic — GitHub Actions runs the whole platform end-to-end on every pull request and every push to main, so a change that breaks the build, the tests, the image, or the compose wiring is caught before it merges, not when I next run docker by hand.

The hard part was not the workflow YAML. It was a question the design note glossed over: when CI brings the compose stack up, where does the `@champion` model come from? The serving container loads `models:/hdb-predictor@champion` at startup and fails fast if it cannot — so without a registered champion, the FastAPI healthcheck never passes and the smoke test cannot run. Locally that registry is the gitignored `mlflow.db` plus `mlruns/`. A fresh CI checkout has neither.

Three facts shaped the decision. The training data is not in CI — `data/` is gitignored and the DVC remote needs MinIO credentials the runner does not hold — so CI cannot retrain the real model. The real champion artifact is 8.4 MB, eleven times over the `check-added-large-files --maxkb=750` pre-commit limit, and committing it would reverse the deliberate decision in commit #40 to keep the registry out of git. But the test suite already trains a tiny synthetic model in `tests/conftest.py`, in well under a second, with no real data.

## Decision

CI seeds its own registry. A new script, `scripts/seed_ci_registry.py`, trains a small `GradientBoostingRegressor` on synthetic HDB rows using the production `build_pipeline` from `training.train`, logs it through local-process MLflow into `./mlflow.db` and `./mlruns`, and points `@champion` at the new version. The workflow then runs the existing `scripts/migrate_artifact_paths_to_proxy.py` to rewrite the stored paths to `mlflow-artifacts:` proxy URIs — the identical two-step flow that produced the real champion in Session B, just with a synthetic model standing in. Both steps run *inside the built FastAPI image*, so the model is pickled with the exact scikit-learn version that will later unpickle it in the serving container; a version skew between seed and serve is impossible by construction.

The workflow has two jobs. `lint-test` runs pre-commit (ruff, ruff-format, mypy) and the full pytest suite. `docker` builds the image with Buildx and the GitHub Actions layer cache, seeds the registry, brings the compose stack up, waits for both healthchecks, and asserts that `POST /predict` returns a response containing `predicted_resale_price`, `model_version`, and `model_alias: champion`. `docker` runs only after `lint-test` passes — there is no point spending three to four minutes on an image build when a unit test is already red. Concurrency is set to cancel in-progress runs on the same ref, so a rapid series of commits on a PR only pays for the latest.

## Consequences

The smoke test proves what matters and nothing it cannot. It exercises the real path — image build, compose networking, the MLflow artifact proxy, model load, SHAP explainer initialisation, and a prediction over HTTP — using a registry built the same way the real one was. It deliberately does not assert a specific price; the synthetic champion predicts something near SGD 495k for the canonical Tampines flat rather than the real model's SGD 586,887, so the test asserts response *shape*, which is the contract a deployment depends on. That the number is synthetic is the honest cost of not having data in CI, and it is the right trade: a shape assertion that runs on every PR is worth far more than an exact-value assertion that cannot run at all.

Runtime is roughly seven to eight minutes cold and three to four minutes warm. The swing is almost entirely the Docker build: the GitHub Actions cache (`type=gha`) turns a three-to-four-minute cold build into about thirty seconds when no layer changed. To make that cache pay off I tag the Buildx output with the exact name compose derives from the project and service — `hdb-mlops-platform-fastapi:latest` — and pin `COMPOSE_PROJECT_NAME`, so `docker compose up` reuses the cached image instead of triggering a second, uncached build of its own.

One implementation note worth recording, because the obvious version is wrong. The health-wait does not grep `docker compose ps --format json` for two `"healthy"` substrings on one line — that output is newline-delimited, one JSON object per service, so the substring match is brittle. CI polls each container's inspected `.State.Health.Status` directly. Because FastAPI's healthcheck only passes once the champion has loaded, waiting on FastAPI implicitly confirms the model load succeeded.

The trade-off I am accepting: the seed couples CI to the shape of the serving pipeline. If `build_pipeline` ever stops exposing the `preprocessor` and `regressor` named steps the SHAP loader reads, the seed and the real model break together — but that is correct, because they *should* break together; the seed is a faithful miniature of the real artifact, not a mock. The script lives in `scripts/` with the other registry tooling and is import-light enough to run inside the slim serving image.

Image push to GHCR is out of scope. CI proves the image builds and runs; it does not publish it, because nothing consumes a published image until there is a deploy target. That is Phase 6 work, tracked in [#32](https://github.com/LEMSingapore/hdb-mlops-platform/issues/32), and it slots in as a final step on the `docker` job — authenticate to GHCR, add `push: true`, tag by SHA — without disturbing anything decided here.

## Alternatives considered

**Commit a registry snapshot to git (Option B).** Force-add `mlflow.db` and the champion's `mlruns/` past the gitignore so CI checks them out directly. The appeal is that the real champion serves the smoke test, so the predicted price matches the README. Rejected on two counts: the champion `model.pkl` is 8.4 MB, which means either raising the `check-added-large-files` limit or excepting the file — both weaken a guard that exists for good reason — and it reverses commit #40's deliberate choice to keep the registry out of git. Committing an 8.4 MB binary blob into history of a repository whose audience is interviewers reading the code is exactly the kind of shortcut this project does not take.

**Retrain the real model in CI (Option A).** Train from scratch on the runner before the compose step. Blocked outright: there is no training data in CI. `data/` is gitignored and the DVC remote needs MinIO credentials the runner does not have, so this option first requires standing up a data-fetch step — credentials, a remote, the lot — and then pays five to ten minutes of training on every run, which would dwarf and largely negate the Buildx caching the design called for. Cost and complexity with no offsetting signal over the synthetic seed.

**Mock the model layer for the smoke test (Option C).** Inject a stub loader so `/predict` returns a canned response without touching MLflow. Cheapest of the three and needs no registry at all. Rejected because it guts the test: the entire point of the compose smoke is to prove the MLflow artifact proxy, the model load, and the SHAP path work together inside containers. A mock asserts that FastAPI can serialise a Pydantic response — which the unit tests already cover — while skipping every integration seam Sessions A and B exist to verify.

**Run the seed on the runner's Python instead of inside the image.** Install the dependencies a second time in the `docker` job and run the seed against that interpreter. Simpler to read, but it reintroduces the version-skew risk the in-image seed eliminates: a model pickled by one scikit-learn build and unpickled by another is a latent failure, and pinning `scikit-learn>=1.5` in `pyproject.toml` does not guarantee two separate installs resolve to the same patch release. Seeding inside the serving image makes the question moot and skips a redundant dependency install.
